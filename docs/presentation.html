Pourquoi l'observabilité?

Sans

  • "Ça marche en local"
  • Logs éparpillés
  • Pas de vue système
  • Debug à l'aveugle en prod

Avec

  • Une seule fenêtre
  • Cause racine en quelques clics
  • Comparaison entre projets
  • SLO mesurables

Objectif PoC

  • Stack open-source
  • Standard OpenTelemetry
  • Reproductible (Docker)
  • Multi-projets / multi-stacks
~1 min. Le "pourquoi", langage métier : "Sans observabilité, quand un client a un problème en production, on debug à l'aveugle — les logs sont éparpillés, on ne voit pas le système. Avec, on trouve la cause racine en quelques clics, on compare nos projets, on mesure la qualité de service. L'objectif du PoC : prouver qu'on peut faire ça avec de l'open-source et un standard, OpenTelemetry."

Les 3 piliers

🔍 Traces

Le voyage d'une requête à travers tous les services. Spans imbriqués, durées, attributs.

📝 Logs

Événements horodatés. Liés au trace_id pour contextualiser.

📈 Metrics

Séries temporelles agrégées. RED (Rate, Errors, Duration), USE, business KPIs.

~1 min. "Trois types de signaux complémentaires. La trace, c'est le voyage d'une requête à travers les services. Le log, c'est un événement horodaté. La métrique, c'est une tendance agrégée dans le temps. Les slides suivantes (traces détaillées, span metrics) sont en option — je les garde pour les questions si on manque de temps."

Traces — exemple

  • User → web → api → RabbitMQ → worker
  • Chaque saut = 1 span (durée, statut, attributs)
  • Tous les spans partagent le même trace_id
  • Vue "waterfall" pour identifier le goulot

Auto-instrumentation .NET ajoute HTTP, EF Core, MassTransit. Python = FastAPI, requests, logging.

Span Metrics — généré automatiquement

Tempo a un metrics-generator qui regarde les traces et produit:

  • traces_spanmetrics_calls_total — RED rate
  • traces_spanmetrics_latency_bucket — RED duration
  • traces_service_graph_request_total — dépendances

→ poussés dans Mimir via remote_write, avec le label project.

Multi-projets, un seul stack

Tag project = resource attribute OTel propagé partout.

OTEL_RESOURCE_ATTRIBUTES="project=poc-telemetry"   # .NET
OTEL_RESOURCE_ATTRIBUTES="project=demo-python"     # Python
  • Variable $project dans les dashboards Grafana
  • Toutes les métriques portent le label project
  • Ajouter un projet = lancer un service avec le bon attribut. Aucune config dans Grafana.
~45 s. Argument d'échelle : "Une seule stack d'observabilité sert plusieurs projets. Chaque signal porte une étiquette 'project'. Dans Grafana, un menu déroulant filtre par projet. Ajouter un nouveau projet ne demande aucune config dans Grafana — juste la bonne étiquette au démarrage."

Dashboards provisionnés

Traces Overview

  • RED metrics par service
  • Heatmap de latence
  • Dernières traces (TraceQL)
  • Service graph dynamique

Error Traces

  • Compteurs + taux d'erreur
  • Top spans en erreur
  • Heatmap latence des erreurs
  • Table filtrée http.status >= 400

Service Graph

Dépendances entre services, dessinées à partir des traces.

  • Nœuds = services qui émettent des spans
  • Arêtes = appels client→server (HTTP, AMQP, gRPC...)
  • Annotations: req/s, latence, taux d'erreur
  • Construit par Tempo metrics-generator, lu par Grafana

Alerting & notifications Livré

🚨 3 règles d'alerte

  • Taux d'erreur global > 5%
  • Latence p95 > 1s (par service)
  • Service down — up == 0

💬 → Discord

  • Contact point + templates
  • 🔴 FIRING / ✅ RESOLVED
  • Route critical plus rapide

Provisionné en YAML, comme les dashboards et datasources — reproductible et versionné. Aucun clic dans l'UI.

~1 min 30. Message clé pour le public mixte : "L'observabilité ne sert pas qu'à regarder après coup — elle nous prévient. On a 3 règles : taux d'erreur, latence, et service indisponible. Quand un seuil est franchi, un message part dans Discord, avec un code couleur rouge quand ça casse et vert quand c'est réglé. Tout est décrit en fichiers (YAML), donc reproductible d'un projet à l'autre." Insister : c'était dans 'à faire', c'est maintenant livré et testé.

Annexe

Détails techniques — pour le Q&A

Difficultés rencontrées · pièges OpenTelemetry · scrape multi-réseaux

Difficultés rencontrées

http2: frame too large

Grafana 11.x active traceQLStreaming par défaut → tente gRPC sur le port HTTP de Tempo.

Fix: GF_FEATURE_TOGGLES_DISABLE=traceQLStreaming

Service graph vide

Requête Prometheus brute ne nourrit pas le panneau nodeGraph.

Fix: datasource Tempo + queryType: serviceMap

HTTP 201 dans la table d'erreurs

Filtre OTel status = error ≠ code HTTP. Exception après réponse → span error mais 201.

Fix: filtrer sur http.response.status_code >= 400

Conventions OTel évoluent

http.method → http.request.method, http.status_code → http.response.status_code.

Fix: queries TraceQL avec les noms récents.

Cibles de scrape dupliquées

  • Les conteneurs sont sur 2 réseaux (app + observability).
  • docker_sd crée une cible par (réseau, port) → doublons à up=0 (réseau non routable depuis le collector).
  • Ces faux up=0 maintenaient l'alerte Service down en firing.

Fix: relabel keep sur le réseau observability (+ port métriques pour RabbitMQ, qui expose plusieurs ports).

À garder pour le Q&A si on demande "comment vous avez câblé le service-down". Sinon ne pas y aller.
